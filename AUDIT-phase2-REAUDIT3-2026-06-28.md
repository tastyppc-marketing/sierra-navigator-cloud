## Blockers

**One blocker — the new HIGH (NEW-A below).** No prior finding is open: all are CLOSED or DEFERRED-ACCEPTED, and the three deferrals (#9 empty-OK, #14 encryption-at-rest, #15 CI hardening) are explicitly accepted for the supervised manual deploy and are NOT blockers.

**NEW-A (HIGH) — Mid-name destructive verbs execute as a guarded `"write"` through the Tier-2 generic caller, with no snapshot and no identity lock.**
- **Residual risk:** `_is_destructive()` (tools_generic.py:96-102) only flags a method when `Delete`/`Deletion` appears *anywhere* OR a destructive verb is in *leading* position. The other 21 verbs in `_DESTRUCTIVE_VERBS` (Release/Disable/Expire/Revoke/Cancel/Archive/… tools_generic.py:37-44) are therefore checked only as a leading token, and "Clean" is absent from the list entirely. Because `_WRITE_VERBS` includes `"Test"` (line 35), three catalogued, reachable endpoints classify as plain `"write"` and route to `guarded_write`→`client.call(path, body, write=True)` (tools_generic.py:214-220), which POSTs verbatim behind only the `allow_write` flag (client.py:86-88): `/voice-and-text-settings.aspx/TestVoiceAndTextReleaseExpiredNumbers` (releases provisioned carrier numbers — irreversible), `…/TestVoiceAndTextManualDisable`, `…/TestVoiceAndTextCleanNumberReferences` (all confirmed present at data/js_bundle_endpoints.json:149-155). These get the write-tier guard (confirm token + audit + 50/session cap) but **not** the recovery snapshot or identity lock the design reserves for destructive ops — directly violating the module invariant "Destruction is categorically impossible via the generic caller" (tools_generic.py:28-29). It is the same mid-name class New-2 closed for `Delete`/`Deletion` only; Release/Disable/Expire/Clean were left open. (One refutation lens voted this within the accepted reviewed-write surface because it is still confirm-gated + audited; two lenses and my own code trace rate it HIGH because the verbs are in the design's *own* destructive list yet escape the destructive branch, and the op is irreversible with no snapshot.)
- **Fix recommendation:** Make `_is_destructive` scan **every CamelCase token** of the method against `_DESTRUCTIVE_VERBS` (split on the uppercase-boundary already implemented in `_starts_with_verb`, refuse if any token equals a destructive verb) rather than only the leading token; and **add "Clean"** (and review for other missing synonyms: Suspend/Detach/Unlink/Wipe/Flush). Avoid a naive substring approach (it would over-refuse reads like `GetArchivedLeads`). Then re-run the catalogue-wide classifier regression to confirm no legitimate write flips to refused, and add a regression test pinning the three `TestVoiceAndText*` paths to `"refused"`. This is a few lines in one function.

## New findings

**NEW-A — HIGH — Tier-2 mid-name destructive-verb misclassification.** Detailed under Blockers. Recommendation: token-wise destructive scan + add "Clean"; regression-pin the voice-and-text endpoints. **Must fix before deploy.**

**NEW-B — MEDIUM — Non-delete writes report a Sierra `responseCode:0` business-rule rejection as a committed success, and record `result="ok"` in the append-only audit log.** `unwrap_response` deliberately PRESERVES a non-empty business-rule `Message` at `responseCode:0` (parsing.py:56-64), and the delete path consumes it via `_assert_delete_ack` (client.py:430/466) — but every non-delete write discards `_call`'s return: `update_content_label` (client.py:279-289), `remove_content_label` (291-296), `update_page_component_title` (328-335), `add/remove_page_content_label_link` (372-388), `remove_page_component_link` (365-370), and `save_html_widget`/`save_content_page`/`save_saved_search` (298-326,390-393). `guarded_write` then treats the non-raising commit as success and writes an immutable `result="ok"` audit row (tools_write.py:181-192). A write Sierra soft-rejected is reported as committed and recorded "ok" in the non-repudiation log — the WRITE-side twin of #9, asymmetric with the delete path that guards it. Real per API_ENDPOINTS.md:179/194 (`{"ResponseCode":0,"Data":null,"Message":"…"}`). **Recommendation:** add an `_assert_write_ack` mirroring `_assert_delete_ack` and call it from each non-delete write method (or have the write methods raise on a returned `Message`/`error`/`success:false`), so `guarded_write`'s existing try/except records `result="error"` instead of `ok`. Strongly recommended before the audit trail is relied on as non-repudiation; not an irreversible-destruction blocker for a watched first deploy.

**NEW-C — LOW — Session-expiry retry re-invokes the non-idempotent pre-delete snapshot, orphaning a `pending-delete` ledger row.** `call_with_refresh` retries the whole delete op once on a logged-out signal (runtime.py:82); the op writes the recovery snapshot (`snapshot_sink(record)`, client.py:425) *before* the remote delete, so if attempt-1 snapshots then gets logged-out at `DeleteContentPage`, attempt-2 writes a second snapshot and `confirm_deletions` flips only the second row to `deleted` (tools_write.py:498-499) — leaving row #1 stranded at `pending-delete` for an already-deleted entity. Irreversible double-delete is correctly prevented (the retry's pre-delete re-fetch fails the id-echo and aborts), so this is ledger-hygiene/accuracy only, not destruction. **Recommendation:** dedupe the snapshot per `(op,row)` — reset captured ledger state before the retried op, or mark the prior `pending-delete` row superseded. Low priority.
